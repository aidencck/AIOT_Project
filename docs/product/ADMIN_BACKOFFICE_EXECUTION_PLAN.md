# 后台管理系统闭环落地执行计划（VP版）

## 1. Premise（前提）
- 当前已具备最小闭环基础链路：规则触发告警、自动建单、工单处理、审计查询、经营看板总览。
- 远程 GitHub Project 已创建后台管理三阶段迭代任务（Phase1-Phase3）。
- 本轮目标是将“可用闭环”升级为“可运行、可巡检、可验收”的最新版本。

## 2. Constraints（约束）
- 现阶段以 Redis 作为闭环状态存储，优先确保运行闭环，不引入重型依赖。
- 保持现有微服务边界不大改：闭环能力继续集中在 `aiot-rule-engine`。
- 本轮必须兼容既有规则动作（`ALERT_LOG` / `WEBHOOK_POST`），避免破坏历史规则。

## 3. Boundaries（边界）
- 包含：技术选型、模块职责、确认人、任务拆解、SLA 超时闭环开发。
- 不包含：复杂 BI 自助报表、分布式工作流引擎、跨团队组织系统改造。

## 4. Endgame（终局）
- 达成“规则触发 -> 告警 -> 工单 -> SLA -> 审计 -> 看板”的可持续运营闭环。
- 每个环节具备可查、可追溯、可干预能力，支持周会运营管理。

## 5. 技术栈选型（本轮确认）
- **后端框架**：Spring Boot 3（现有栈一致，避免迁移成本）。
- **事件与状态存储**：Redis Stream + Redis Hash（延续现有事件链路，低成本快速落地）。
- **接口风格**：REST + 统一 `Result` 包装（复用全局返回规范）。
- **可观测性**：MDC TraceId + 审计记录（支撑问题追踪）。
- **调度机制**：Spring Scheduling（实现 SLA 自动巡检）。

## 6. 模块职责与确认人
- **模块：规则执行与告警**
  - 职责：规则命中后创建告警并驱动后续闭环。
  - 选型确认人：`待 VP 指定`
  - 技术确认人：`待 Tech Lead 指定`
- **模块：工单与 SLA**
  - 职责：自动建单、认领/完结、超时检测与升级。
  - 选型确认人：`待 VP 指定`
  - 技术确认人：`待 Tech Lead 指定`
- **模块：审计与看板**
  - 职责：审计记录、运营核心指标聚合输出。
  - 选型确认人：`待 VP 指定`
  - 技术确认人：`待 Tech Lead 指定`

## 7. 任务拆解（开发顺序）
- **任务 A：SLA 模型补齐**
  - 增加工单超时相关字段（到期时间、是否超时）。
- **任务 B：SLA 巡检能力**
  - 增加手动巡检接口 + 定时巡检任务。
  - 超时后更新工单状态并写审计事件。
- **任务 C：看板指标增强**
  - 增加超时工单数等运营关键指标。
- **任务 D：远程项目回写**
  - 将本轮开发结果同步到对应 GitHub Issue，进入待验收状态。

## 8. 验收口径（本轮）
- 当工单超出 SLA 时，系统能够自动识别并标记为超时。
- 支持手动触发 SLA 巡检，返回本次处理数量。
- 看板可展示超时工单相关指标。
- 审计中可追溯 SLA 超时处置事件。
