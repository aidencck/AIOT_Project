## 2. 下一步（必须完成）执行项

### 2.1 Sprint-S1：通信安全与契约硬化（本周）
- 目标：
  - 所有 `/api/v1/internal/**` 仅接受服务令牌或 mTLS 身份。
  - 所有跨服务响应统一按 `Result<T>` 解析并校验 `code/data`。
- 交付：
  - 内部接口鉴权开关与环境配置文档。
  - 契约测试脚本和 CI guard 通过记录。
- 验收：
  - 未携带服务令牌访问内部接口返回 `401`。
  - 契约脚本 100% 通过。

### 2.2 Sprint-S2：一致性补偿可审计化（下周）
- 目标：
  - `home/room` 删除后的设备解绑补偿具备“可重试 + 可追踪”能力。
- 交付：
  - 补偿执行日志字段规范（traceId、homeId/roomId、result、latency）。
  - 巡检 SQL 与定时任务（检查悬挂 homeId/roomId）。
- 验收：
  - 灰度环境连续 3 天无新增悬挂关联。
  - 补偿失败可在日志/告警中定位到具体请求。

### 2.3 Sprint-S3：容错与可观测补齐（两周内）
- 目标：
  - 核心同步调用全部具备超时、重试、熔断/降级策略。
  - 关键链路具备统一调用指标与告警。
- 交付：
  - 统一 `WebClient` 配置（timeout/retry）。
  - 接口层 Prometheus 指标（调用数、错误数、耗时分位数）。
- 验收：
  - 下游故障注入场景下，主链路可控退化，不出现级联雪崩。

## 3. 执行清单（DoD）

- 每个优化项必须同时满足：
  - 代码改动 + 配置改动 + 文档更新
  - 集成脚本可执行并通过
  - CI Guard 通过
  - 回滚方案明确（配置回滚或开关回退）

## 4. 风险与应对

- 风险：内部令牌泄漏导致横向调用风险。  
  应对：短周期轮换、密钥托管、后续迁移 mTLS。
- 风险：同步补偿在下游抖动时阻塞删除流程。  
  应对：增加重试上限与异步补偿兜底任务。
- 风险：测试环境数据不稳定导致脚本波动。  
  应对：引入固定种子数据与测试专用命名空间。
