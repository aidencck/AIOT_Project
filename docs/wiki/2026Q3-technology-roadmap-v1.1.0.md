# 2026Q3 技术路线图（版本化）v1.1.0

状态：`planned`
版本：`v1.1.0`
周期：`2026Q3`
聚焦主题：`压测体系 + 瓶颈治理 + 技术债收敛`

## Premise / Constraints / Boundaries / Endgame

- Premise：
  - 设备状态链路已具备 Redis Stream + Consumer Group + DLQ + 批量刷盘基础能力。
  - 当前主要短板是“压测口径可复现但环境不稳定、端到端指标不足、发布前缺少性能门禁”。
- Constraints：
  - 现阶段以 Compose 单节点验证为主，容量结论必须区分“单节点上限”与“目标态扩展上限”。
  - 业务迭代与治理任务并行，需按版本纳入 DoD，避免治理被功能挤占。
- Boundaries：
  - 本路线图只覆盖性能与架构演进，不包含产品功能扩展需求。
- Endgame：
  - 将性能治理从“专项动作”升级为“版本标准动作”：每个版本都必须完成压测、归因、回归与门禁。

## Q3 版本节奏

| 版本 | 时间窗 | 目标 | 核心交付 | 验收标准 |
| --- | --- | --- | --- | --- |

## 里程碑任务拆解

### v1.1.0（M1）：基线体系固化

- 任务1：压测脚本标准化
  - 完成状态码分布、`p50/p95/p99`、报告留档能力。
- 任务2：环境门禁脚本化
  - 压测前自动检查服务健康、关键配置、依赖可达。
- 任务3：基线报告版本化
  - 报告结构固定为“场景-数据-结论-风险-动作”。
- DoD：
  - 基线报告可审计（命令、参数、时间戳、原始输出齐全）。

### v1.2.0（M2）：核心瓶颈治理

- 任务1：消费与落库可靠性增强
  - 评估并落地“ACK 时机后移”或“持久化缓冲”方案，降低 ACK 早于持久化风险。
- 任务2：刷盘策略演进
  - 从“按状态多 SQL”升级为 `CASE WHEN` 单 SQL 批量更新。
- 任务3：端到端可观测性
  - 增加 webhook 入站到落库完成的全链路时延指标。
- DoD：
  - 高并发下 `pending` 不持续上升，`flush.failed` 受控。

### v1.3.0（M3）：工程化收敛

- 任务1：CI 性能回归阶段
  - 将 Smoke + Load 压测接入流水线，阈值失败自动阻断发布。
- 任务2：容量模型
  - 输出单节点容量基线与目标态扩容建议（实例数、Redis/MySQL压力区间）。
- 任务3：技术债结项机制
  - 每个版本设置固定治理点位并在迭代复盘中核销。
- DoD：
  - 发布具备“功能通过 + 稳定性通过 + 性能通过”三门禁。

## 技术债治理主线

- 主线A：吞吐与时延
  - 入口限流、消费批处理、刷盘 SQL 形态优化。
- 主线B：可靠性
  - ACK 策略、pending 回收、DLQ 追踪与回放闭环。
- 主线C：可观测与回归
  - 指标标准化、压测自动化、阈值门禁化。

## 风险与预案

| 风险 | 触发信号 | 预案 |
| --- | --- | --- |
| 环境波动导致数据不可比 | 同参数结果离散度过高 | 固定压测时窗、固定依赖版本、固定数据集 |
| 局部优化引发瓶颈迁移 | 入口改善后 DB/Redis 指标恶化 | 使用端到端看板按链路验收 |
| 治理任务被业务需求挤压 | 版本治理项延期 | 将治理项写入版本 DoD 与发布门禁 |

## 北极星指标（Q3）

- 可用性：状态上报链路 `2xx >= 99.9%`
- 性能：`p95/p99` 季度内持续下降并稳定在阈值内
- 稳定性：`flush.failed`、`DLQ`、`pending` 指标异常可定位、可追踪、可回放
- 工程化：每个版本必须有性能回归记录与结论归档
