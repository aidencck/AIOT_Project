# 2026Q3 技术路线图（版本化）v1.0.0

状态：`draft-approved-ready`
版本：`v1.0.0`
周期：`2026Q3`
目标链路：`设备状态上报主链路（可复用到事件主干）`

## Premise / Constraints / Boundaries / Endgame

- Premise：
  - 现网主链路已具备 Redis Stream + Consumer Group + DLQ 的可靠性底座。
  - 当前阶段核心矛盾是“可观测性深度不足 + 压测自动化不足 + 高峰下吞吐余量不明确”。
- Constraints：
  - 生产形态仍以 Compose/单节点验证为主，容量结论需区分“单节点上限”和“目标态集群上限”。
  - 团队需在新增业务交付与技术债治理之间平衡节奏。
- Boundaries：
  - 本路线图聚焦“性能与架构演进”，不覆盖产品功能需求扩展。
  - 仅规划后端与平台能力，不覆盖终端固件升级节奏。
- Endgame：
  - 形成“指标驱动的版本化性能治理机制”，将压测、优化、回归纳入每个版本 DoD。

## 里程碑拆解（按月）

### M1：基线与可观测统一（v1.0.0）

- 交付1：固化压测脚本 `perf_device_status_webhook.sh` 并形成统一参数模板。
- 交付2：补齐状态刷盘关键指标并接入 Prometheus/Grafana 面板。
- 交付3：形成《性能基线报告 v1.x》模板（链路、口径、结果、结论统一）。
- DoD：
  - 任一环境可在 30 分钟内复现压测并输出报告。
  - 压测后能回答“哪里慢、慢多少、为何慢”。

### M2：核心瓶颈治理（v1.1.0）

- 交付1：入口保护（Webhook 限流/突刺防护）和消费端批处理策略优化。
- 交付2：端到端时延指标（入站时间 -> 消费时间 -> 落库时间）串联。
- 交付3：DLQ 分析与自动回放工具（低风险场景）。
- DoD：
  - 负载突增时 `5xx` 不出现连续爬升。
  - `pending` 不持续累积，DLQ 可解释且可追踪处理。

### M3：工程化与发布门禁（v1.2.0）

- 交付1：CI/CD 增加性能回归阶段（烟囱压测 + 指标阈值判定）。
- 交付2：容量模型（单节点与目标态集群双模型）与扩缩容建议。
- 交付3：发布门禁（指标阈值不达标禁止上线）。
- DoD：
  - 版本发布具备“功能 + 稳定性 + 性能”三重门禁。
  - 路线图指标可周度追踪，异常项可责任到模块。

## 技术债治理主线

- 主线A：吞吐治理
  - Webhook 入站保护、消费批处理、DB 批量写策略演进。
- 主线B：可观测治理
  - 自定义指标标准化、Trace 贯通、压测报表自动生成。
- 主线C：工程化治理
  - 压测场景脚本化、结果版本化、回归自动化与阈值门禁。

## 风险与预案

- 风险1：环境依赖不稳定导致压测数据不可比。
  - 预案：固定种子数据 + 固定环境参数 + 固定压测时窗。
- 风险2：仅优化局部导致“瓶颈迁移”到其他组件。
  - 预案：坚持端到端指标看板，版本验收按链路而非单点。
- 风险3：功能迭代挤压治理节奏。
  - 预案：将性能治理纳入版本 DoD，不再作为可选项。

## 指标北极星（Q3）

- 北极星1：设备状态链路 2xx 成功率持续稳定（压测与灰度双场景）。
- 北极星2：链路 p95 延迟和 `flush.failed` 指标可持续下降。
- 北极星3：每个版本发布前均有可追溯性能回归报告（版本化存档）。
