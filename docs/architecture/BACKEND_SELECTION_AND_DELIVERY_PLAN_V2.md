# 后端技术框架选型与开发落地方案（家庭/成员/产品/设备/OTA）

## Premise
- 目标是支撑后台管理五域：家庭、成员、产品、设备、OTA 的可运行闭环。
- 现有工程已采用 Spring Boot + MyBatis-Plus + Redis + Nacos，家庭/成员/产品/设备已具备基础能力。
- 本次重点补齐 OTA 业务域，并统一五域后端边界和协作机制。

## Constraints
- 保持现有微服务拆分，不做大规模重构与技术替换。
- 所有接口沿用统一异常和响应规范，遵循软删除和审计可追踪要求。
- OTA 首期采用最小可运行方案，优先可交付与可扩展。

## Boundaries
- 包含：后端技术选型、模块职责、数据模型、接口分层、最小开发清单。
- 不包含：前端页面改版、复杂流程编排引擎、多云多活和跨区域容灾设计。

## Endgame
- 实现“产品定义 -> 设备入网 -> 家庭归属与成员授权 -> OTA 升级 -> 结果回写与审计”的完整链路。
- 形成稳定的后端标准：可扩展、可观测、可回滚、可验收。

## 技术栈选型（定稿）
- 应用框架：Spring Boot 3.x + Spring MVC。
- 服务治理：Spring Cloud Alibaba Nacos（注册与发现）。
- 持久层：MyBatis-Plus（实体映射、分页、逻辑删除）。
- 关系数据库：MySQL（主数据、任务数据、升级记录）。
- 缓存与事件：Redis（状态缓存、流式事件、短期去重）。
- 鉴权与权限：JWT 用户态鉴权 + `X-Internal-Token` 内部接口鉴权 + 家庭粒度权限校验。
- 可观测：MDC TraceId + 业务审计日志。

## 模块边界与职责
- `aiot-home-service`
  - 家庭主数据、成员关系、角色权限（Owner/Admin/Member）。
- `aiot-device-service`
  - 产品、设备、配网、OTA 主体能力（本次新增 OTA）。
- `aiot-auth-service`
  - 设备接入认证、上下线事件源。
- `aiot-rule-engine`
  - 规则执行、告警工单闭环，与设备事件联动。

## 五域开发状态
- 家庭：已具备创建/查询/删除、家庭内权限校验。
- 成员：已具备增删改查与角色控制。
- 产品：已具备创建、查询、物模型更新。
- 设备：已具备 CRUD、状态流转、配网与心跳更新。
- OTA：本次补齐固件包管理、升级任务、升级记录、结果上报。

## 本轮开发清单（已落地）
- OTA 数据模型：
  - `firmware_package`
  - `ota_upgrade_task`
  - `ota_upgrade_record`
- OTA 接口能力：
  - 发布固件包
  - 固件包列表查询
  - 创建升级任务（按家庭、产品、设备集合）
  - 任务查询与任务列表查询
  - 设备升级结果上报（内部鉴权）
- 权限策略：
  - 创建/查询 OTA 任务前校验家庭权限。
  - 设备上报升级结果走内部接口 + `X-Internal-Token`。

## 迭代建议
- Iteration-1（当前）：OTA 最小可运行闭环（已完成）。
- Iteration-2：灰度策略、批次暂停/恢复、失败重试策略。
- Iteration-3：失败率熔断、自动回滚、升级可视化看板。
